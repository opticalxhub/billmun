The login bug after self-approval is almost certainly one of these three things — check in this order:

Most likely cause — password not being hashed on registration
When you create the account it stores the plain text password. When NextAuth tries to verify it, bcryptjs compares the plain text input against what it thinks is a hash and it fails. Check your registration API route — make sure it does bcrypt.hash(password, 12) before saving to the database and that the field being saved is the hashed result, not the raw input.

Second likely cause — NextAuth credentials provider comparing wrong fields
In your NextAuth config the authorize function probably does something like bcrypt.compare(password, user.password) — make sure the field name matches exactly what Prisma returns. If your schema field is passwordHash but the authorize function reads user.password it always returns null and NextAuth rejects the login silently.

Third likely cause — status check blocking the session
If your authorize function checks if (user.status !== "APPROVED") return null and the approval flow has a bug where it sets the status string slightly differently — like "approved" lowercase instead of the enum value "APPROVED" — the check fails even though the user looks approved in the database. Open your database directly and check the exact value stored in the status column after you approve yourself.

For the /911 emergency route — add this to your prompt:
Build a hidden emergency access route at /911. This route is not linked anywhere in the application — it exists only for developer access during testing. Visiting /911 renders a minimal full-screen dark page on #080808 with just the BILLMUN wordmark and a single input field asking for an emergency passphrase. The passphrase is stored only in the environment variable EMERGENCY_PASSPHRASE and never in the database or client code. Submitting the correct passphrase creates a temporary session valid for exactly 2 minutes granting EXECUTIVE_BOARD level access. This session is stored in the database in a new EmergencySession table with id, createdAt, and expiresAt. On every protected route, middleware checks for this emergency session token in addition to the normal NextAuth session — if the emergency token exists and has not expired, access is granted. After 2 minutes the session is automatically invalidated and any further requests redirect to login. The /911/dash route redirects directly to /dashboard using the emergency session. A visible countdown timer in the top-right corner of every page shows the remaining emergency session time in minutes and seconds — when it hits zero the page automatically redirects to /login and clears the session. This route must be completely removed or disabled before any production deployment by checking a DISABLE_EMERGENCY_ACCESS environment variable that if set to true returns a 404 on any /911 request.
